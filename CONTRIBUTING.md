# 기여 가이드

Full Stack FastAPI 템플릿에 기여해 주셔서 감사합니다! 🙇

## 먼저 토론(Discussion)부터

**큰 변경 사항**(새 기능, 아키텍처 변경, 대규모 리팩토링)의 경우, 먼저 [GitHub Discussion](https://github.com/fastapi/full-stack-fastapi-template/discussions)을 열어 주세요. 이렇게 하면 구현에 많은 시간을 투자하기 전에 커뮤니티와 메인테이너로부터 접근 방식에 대한 피드백을 받을 수 있습니다.

작고 단순한 변경 사항의 경우, 토론 없이 바로 Pull Request를 제출해도 됩니다. 여기에는 다음이 포함됩니다:

- 오타 및 문법 수정
- 작고 재현 가능한 버그 수정
- 린트 경고 또는 타입 오류 수정
- 사소한 코드 개선 (예: 사용하지 않는 코드 제거)

## 개발 환경

개발 환경 설정, 스택 실행, 린팅, pre-commit 훅 등에 대한 자세한 내용은 [개발 가이드](development.md)를 참고하세요.

## Pull Request

Pull Request를 제출할 때:

1. 제출 전에 모든 테스트가 통과하는지 확인하세요.
2. PR은 하나의 변경 사항에 집중하세요.
3. 기능을 변경하는 경우 테스트도 업데이트하세요.
4. PR 설명에 관련 이슈를 참조하세요.

## 자동화된 코드와 AI

작업을 수행하고 최대한 효율적으로 기여하기 위해 AI(LLM) 도구 등 원하는 모든 도구를 사용하는 것을 권장합니다. 그러나 기여에는 의미 있는 인간의 개입, 판단, 맥락 등이 포함되어야 합니다.

PR에 들어간 **인간의 노력**(예: LLM 프롬프트 작성)이 **우리가 리뷰하는 데 필요한 노력**보다 **적다면**, PR을 **제출하지 마세요**.

이렇게 생각해 보세요: 우리도 이미 LLM 프롬프트를 작성하거나 자동화 도구를 직접 실행할 수 있고, 그것이 외부 PR을 리뷰하는 것보다 빠릅니다.

### 자동화된 AI PR 닫기

AI가 생성했거나 유사한 방식으로 자동화된 것으로 보이는 PR은 플래그를 달고 닫겠습니다.

댓글과 설명도 마찬가지입니다. LLM이 생성한 내용을 복사하여 붙여넣지 마세요.

### 인적 노력에 대한 서비스 거부 공격

자동화 도구와 AI를 사용하여 우리가 신중하게 검토하고 처리해야 하는 PR이나 댓글을 제출하는 것은 우리의 인적 노력에 대한 [서비스 거부 공격](https://en.wikipedia.org/wiki/Denial-of-service_attack)과 같습니다.

PR을 제출하는 사람의 노력은 매우 적지만(LLM 프롬프트 하나), 우리 쪽에서는 많은 노력(코드를 신중하게 검토)이 발생합니다.

그렇게 하지 마세요.

반복적으로 자동화된 PR이나 댓글로 스팸을 보내는 계정은 차단해야 합니다.

### 도구를 현명하게 사용하세요

벤 삼촌이 말했듯이:

> 큰 ~~힘~~**도구**에는 큰 책임이 따른다.

의도치 않게 해를 끼치는 것을 피하세요.

여러분 손에는 놀라운 도구가 있습니다. 효과적으로 돕기 위해 현명하게 사용하세요.
